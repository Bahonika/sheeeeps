# Деплой Sheeeeps — общее пастбище онлайн (Yandex Cloud)

Пошаговая инструкция уровня «первый деплой в жизни». В конце у тебя будет публичная
ссылка, по которой любой откроет игру в браузере и попадёт на живое пастбище, плюс
понимание, как это обновлять и чинить.

Вся игра размещается на **одной виртуальной машине в Yandex Cloud** — и сервер
(симуляция овец), и клиент (сайт игры) за одним доменом с TLS. Так надёжнее всего для
игроков из России и всё в одном месте. Ориентировочная стоимость — **порядка
1.5–3 тыс. ₽/мес** (см. [раздел 10](#10-сколько-это-стоит)), с большим запасом под
твой бюджет 18 000 ₽.

---

## 0. Из чего состоит игра

- **Сервер** — постоянный процесс, круглосуточно считает симуляцию овец и рассылает
  клиентам снапшоты по WebSocket. Это тот же симуляционный код, что и в одиночной игре,
  только без картинки. Запускается в Docker-контейнере на ВМ.
- **Клиент** — статические файлы (собранный из Flutter сайт), никакой логики игры.
  Раздаются с той же ВМ.
- **Caddy** — маленький веб-сервер перед ними: отдаёт клиент, проксирует WebSocket на
  сервер и **сам получает бесплатный TLS-сертификат**, чтобы всё работало по `https`/
  `wss` (браузер не пускает игру по незашифрованному соединению).

Схема: `браузер → https/wss → Caddy (443) → статика клиента + WebSocket на localhost:8080`.

## 1. Что понадобится

**Аккаунты и доступы:**
- [Yandex Cloud](https://console.yandex.cloud) с привязанным платёжным аккаунтом (у тебя
  уже есть).
- [GitHub](https://github.com/join) — код уже лежит в `github.com/Bahonika/sheeeeps`.
- Бесплатный поддомен для TLS — заведём на [DuckDNS](https://www.duckdns.org) в шаге 3.3
  (или свой домен, если есть).

**Локально установить:**
- [Git](https://git-scm.com/downloads) и [Flutter SDK](https://docs.flutter.dev/get-started/install)
  (включает Dart) — нужны, чтобы собрать клиент. Проверка: `flutter --version`.
- SSH-ключ. Если нет — создай: `ssh-keygen -t ed25519` (жми Enter на все вопросы).
  Публичный ключ будет в `~/.ssh/id_ed25519.pub` — его содержимое понадобится при
  создании ВМ.

## 2. Проверка локально (перед облаком)

Убедимся, что всё собирается и работает на твоём компьютере.

**Сервер:**
```bash
cd путь/к/sheeeeps
dart run bin/server.dart
```
Успех, если появилось:
```
[…] pasture server up — listening on 0.0.0.0:8080 (protocol v2, 450 sheep, up to 16 shepherds)
```
В другом терминале: `curl http://localhost:8080/health` → `ok`. Оставь сервер запущенным.

**Клиент (в новом терминале):**
```bash
flutter run -d chrome -t lib/main_pasture.dart --dart-define=PASTURE_URL=ws://localhost:8080
```
Введи имя → «Играть» → появишься собакой на пастбище. Загони всех овец — увидишь
фейерверк и таймер раунда. Работает локально — значит дальше только вопрос хостинга.
Останови оба процесса (`Ctrl+C`).

## 3. Создать виртуальную машину в Yandex Cloud

### 3.1. Создание ВМ

1. [console.yandex.cloud](https://console.yandex.cloud) → выбери свой каталог (folder) →
   сервис **Compute Cloud** → **Создать ВМ**.
2. **Имя**: `pasture-server`. **Зона доступности**: `ru-central1-a`.
3. **Образ/загрузочный диск**: вкладка **Ubuntu** → `Ubuntu 22.04 LTS`. Диск: **SSD
   20 ГБ** (можно HDD — дешевле).
4. **Вычислительные ресурсы**: платформа Intel Ice Lake; **vCPU = 2**, **гарантированная
   доля = 20 %** (нашему лёгкому серверу хватает; можно 50 % для запаса), **RAM = 2 ГБ**.
   Галочку **«Прерываемая»** НЕ ставь — сервер должен работать 24/7 (прерываемую Yandex
   может остановить в любой момент).
5. **Сетевые настройки**: подсеть в `ru-central1-a`. **Публичный адрес** → выбери
   **Автоматически** (позже можно зарезервировать статический, чтобы IP не менялся).
6. **Доступ**: **Логин** = `ubuntu`; в **SSH-ключ** вставь содержимое своего
   `~/.ssh/id_ed25519.pub`.
7. **Создать ВМ**. Через ~минуту в списке появится **публичный IP** — запиши его.

### 3.2. Открыть порты 80 и 443

По умолчанию группа безопасности пускает только SSH (22). Добавим HTTP/HTTPS:

1. **Virtual Private Cloud** → **Группы безопасности** → открой ту, что привязана к ВМ
   (или создай новую и привяжи к сетевому интерфейсу ВМ).
2. **Добавить правило** (входящий трафик), по одному на порт:
   - `Порт 22`, протокол TCP, источник `CIDR 0.0.0.0/0` — SSH (если ещё нет).
   - `Порт 80`, TCP, `0.0.0.0/0` — для выпуска TLS-сертификата.
   - `Порт 443`, TCP, `0.0.0.0/0` — сама игра (https/wss).
3. Исходящий трафик — разрешён по умолчанию, ничего не трогай.

> Если раздела групп безопасности в твоей сети нет (он бывает в Preview) — значит весь
> трафик и так разрешён, шаг можно пропустить.

### 3.3. Бесплатный домен для TLS

Let's Encrypt не выдаёт сертификат на голый IP, нужен домен. Проще всего — бесплатный
поддомен на [DuckDNS](https://www.duckdns.org): войди через GitHub/Google, придумай имя
(например `sheeeeps`), впиши в поле **current ip** публичный IP своей ВМ → **update ip**.
Получишь адрес вида `sheeeeps.duckdns.org`. Дальше в инструкции подставляй свой.

## 4. Настроить сервер на ВМ

Зайди по SSH (подставь свой ключ и IP):
```bash
ssh -i ~/.ssh/id_ed25519 ubuntu@ПУБЛИЧНЫЙ_IP
```

На самой ВМ по очереди:

```bash
# 1. Docker
curl -fsSL https://get.docker.com | sudo sh

# 2. Caddy (даёт бесплатный TLS автоматически)
sudo apt update && sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update && sudo apt install -y caddy git

# 3. Собрать и запустить сервер игры в контейнере
git clone https://github.com/Bahonika/sheeeeps.git
cd sheeeeps
sudo docker build -t pasture-server .            # ~5–10 минут в первый раз
sudo docker run -d --name pasture --restart=always -p 127.0.0.1:8080:8080 pasture-server
```

`--restart=always` — это супервизор из ТЗ: контейнер сам поднимется после падения или
перезагрузки ВМ. Проверка: `curl http://localhost:8080/health` → `ok`.
(`-p 127.0.0.1:8080` — сервер слушает только локально, наружу его отдаёт Caddy по TLS.)

Клиент положим позже в `/var/www/pasture`, подготовим папку:
```bash
sudo mkdir -p /var/www/pasture
```

Настроим Caddy (замени домен на свой из 3.3):
```bash
sudo nano /etc/caddy/Caddyfile
```
Впиши целиком (шаблон также лежит в репозитории в `deploy/Caddyfile`):
```
sheeeeps.duckdns.org {
	# WebSocket-апгрейды уходят на игровой сервер…
	@ws {
		header Connection *Upgrade*
		header Upgrade    websocket
	}
	handle @ws {
		reverse_proxy localhost:8080
	}
	handle /health {
		reverse_proxy localhost:8080
	}
	# …всё остальное — статика клиента.
	handle {
		root * /var/www/pasture
		try_files {path} /index.html
		file_server
	}
}
```
Применить:
```bash
sudo systemctl restart caddy
```

**Проверка:** открой в браузере `https://sheeeeps.duckdns.org/health` — должно написать
`ok` и показать валидный замок TLS (первый запрос может подумать несколько секунд, пока
Caddy получает сертификат). Сервер готов. Адрес для клиента — `wss://sheeeeps.duckdns.org`.

## 5. Собрать и выложить клиент

Клиент кладём на ту же ВМ. Собираем **локально** (на своём компьютере), указывая адрес
сервера, и копируем результат на ВМ.

**На своём компьютере:**
```bash
# Linux/macOS:
scripts/build-web.sh wss://sheeeeps.duckdns.org
# Windows PowerShell:
.\scripts\build-web.ps1 wss://sheeeeps.duckdns.org
```
Появится папка `build/web` — готовый сайт с зашитым адресом сервера. Скопируй её на ВМ:
```bash
scp -i ~/.ssh/id_ed25519 -r build/web/* ubuntu@ПУБЛИЧНЫЙ_IP:/tmp/web/
```
**На ВМ** переложи файлы в папку, которую отдаёт Caddy:
```bash
sudo rm -rf /var/www/pasture/*
sudo cp -r /tmp/web/* /var/www/pasture/
```

**Готово.** Открой `https://sheeeeps.duckdns.org` — увидишь экран ввода имени. Введи имя
→ «Играть» → появишься на пастбище (сверху таймер раунда и рекорд дня, справа таблица
«Загнали»). **Эта ссылка и есть публичный адрес игры — делись ей с друзьями.**

> **Альтернатива (если не хочешь копировать клиент на ВМ):** можно выложить папку
> `build/web` на бесплатный [Cloudflare Pages](https://dash.cloudflare.com) (Workers &
> Pages → Create → Pages → Upload assets, перетащить содержимое `build/web`). Тогда
> клиент будет на `https://…pages.dev`, а сервер — на ВМ. Минус: Cloudflare в России
> иногда работает нестабильно, поэтому по умолчанию держим всё на ВМ.

## 6. Как обновлять после правок кода

**Сервер** (правки симуляции/сети):
```bash
ssh -i ~/.ssh/id_ed25519 ubuntu@IP
cd sheeeeps && git pull
sudo docker build -t pasture-server .
sudo docker rm -f pasture && sudo docker run -d --name pasture --restart=always -p 127.0.0.1:8080:8080 pasture-server
```

**Клиент** (правки интерфейса):
```bash
# локально пересобрать и снова скопировать:
scripts/build-web.sh wss://sheeeeps.duckdns.org
scp -i ~/.ssh/id_ed25519 -r build/web/* ubuntu@IP:/tmp/web/
# на ВМ:
sudo rm -rf /var/www/pasture/* && sudo cp -r /tmp/web/* /var/www/pasture/
```
Игрокам после обновления клиента, возможно, понадобится **Ctrl+F5** (сброс кэша).

> Сменил домен? Пересобери **и** клиент с новым `PASTURE_URL`, иначе он будет стучаться
> на старый адрес.

## 7. Логи и перезапуск

```bash
sudo docker logs -f pasture        # логи сервера в реальном времени
sudo docker restart pasture        # перезапустить сервер
sudo systemctl restart caddy       # перезапустить веб-слой (TLS/раздача)
journalctl -u caddy -f             # логи Caddy (если проблемы с TLS)
```

Что пишет сервер (по ТЗ): старт; каждое подключение/отключение (`join`/`leave`);
завершение раунда с временем и рекордом дня; и раз в 10 секунд — перцентили времени тика
и число игроков:
```
[…] tick p50=0.18ms p95=0.33ms p99=0.51ms max=1.2ms n=600 players=7
```
Тик держится сильно ниже 10 мс даже при 16 собаках и 500 овцах (замерено: p99 ≈ 0.4 мс).

## 8. Если не работает

- **`https://домен/health` не открывается / нет замка.** TLS ещё не выпущен или закрыт
  порт. Проверь: (1) в группе безопасности открыты **80 и 443** (шаг 3.2); (2) DuckDNS
  указывает на актуальный IP ВМ; (3) `journalctl -u caddy -f` — Caddy пишет, почему не
  смог получить сертификат (частая причина — закрыт порт 80).

- **Игра не подключается («Подключение к пастбищу…» висит).** (1) `curl
  http://localhost:8080/health` на ВМ должен давать `ok` — если нет, смотри `docker logs
  pasture`; (2) клиент собран с точным `wss://твой-домен` (пересобери `build-web`); (3)
  в консоли браузера (F12) не должно быть `Mixed Content` — адрес именно `wss://`.

- **`wss` не коннектится, хотя `/health` по https отвечает.** Проверь блок `@ws` в
  `Caddyfile` (матч по заголовкам Upgrade) и `sudo systemctl restart caddy`.

- **Docker-сборка падает на нехватке памяти.** 2 ГБ обычно хватает, но если упало —
  временно поставь ВМ 4 ГБ RAM на время сборки, потом верни 2 ГБ (или собери образ один
  раз и не пересобирай без нужды).

- **Процесс «умер» после выхода из SSH.** Так и должно быть только если запускал вручную.
  Мы запускаем через `docker run -d --restart=always` — он живёт отдельно от SSH-сессии и
  сам поднимается после перезагрузки ВМ.

- **Браузер показывает старый клиент после обновления.** Кэш: **Ctrl+F5**
  (Cmd+Shift+R на macOS).

- **Высокий пинг у тебя из Сербии.** Это нормально: сервер в Москве (`ru-central1`), для
  друзей из РФ пинг минимальный, у тебя ~50–70 мс. Клиент сглаживает движение
  интерполяцией, так что играется комфортно.

## 9. Безопасность и устойчивость (уже в коде)

Сервер сам защищён по ТЗ, отдельно настраивать не нужно: мусорные сообщения от клиента
молча отбрасываются, исключение в одном соединении не роняет процесс, вход ограничен
20 командами/сек с клиента, при 16 подключённых — новый получает вежливый отказ
«Пастбище переполнено». Собака засыпает после 3 минут без ввода и отключается через 10,
освобождая слот. При нуле игроков сервер снижает частоту симуляции до 2 тик/с и мгновенно
возвращает полную при первом заходе.

## 10. Сколько это стоит

Ты платишь Yandex Cloud только за то, что реально работает. Порядок цифр (проверяй в
[калькуляторе Yandex Cloud](https://yandex.cloud/ru/prices) — тарифы меняются):

- **ВМ** 2 vCPU (20 % гарантированной доли) + 2 ГБ RAM, непрерываемая, 24/7 — Compute
  Cloud стартует примерно от **~1 660 ₽/мес**.
- **Диск** SSD 20 ГБ — ещё несколько сотен ₽/мес (HDD дешевле).
- **Публичный IP** и **исходящий трафик** — копейки при нашей нагрузке (снапшот ≈ 2.7 КБ
  × 20/с ≈ 55 КБ/с на игрока; при пустом пастбище трафик нулевой).

Итого ориентировочно **1.5–3 тыс. ₽/мес** — глубоко в пределах бюджета 18 000 ₽.

**Чтобы не было сюрпризов:** в консоли → **Billing (Биллинг)** → **Бюджеты** → создай
бюджет с уведомлением, например на 5 000 ₽/мес. Yandex пришлёт письмо при приближении к
порогу. Дешевле держать ВМ поменьше (20 % доли CPU) — нашему серверу этого хватает
(замер: тик p99 ≈ 0.4 мс).
