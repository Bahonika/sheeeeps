# Деплой Sheeeeps — общее пастбище онлайн (Stage 3)

Это пошаговая инструкция «первый деплой в жизни». Дочитав до конца, ты получишь
публичную ссылку, по которой любой человек открывает игру в браузере и попадает
на живое пастбище, плюс понимание, как это обновлять и чинить.

> **Важно про деньги.** Всё ниже — бесплатно в разумных пределах трафика.
> Конкретные лимиты и что будет при их превышении — в разделе
> [«Бесплатные лимиты и трафик»](#9-бесплатные-лимиты-и-трафик). Кратко: для
> проверки гипотезы «людям весело 15 минут» бесплатного тарифа хватает с запасом.

---

## 0. Из чего состоит игра

Две независимые части:

1. **Клиент** — это статические файлы (HTML/JS/CSS), собранные из Flutter. Никакой
   логики игры в них нет. Их кладут на бесплатный CDN-хостинг статики
   (**Cloudflare Pages**). Раздаётся по `https://…`.
2. **Сервер** — один постоянный процесс, который круглосуточно считает симуляцию
   овец и рассылает клиентам снапшоты по WebSocket. Это тот же симуляционный код,
   что и в одиночной игре, только без картинки. Его поднимают на бесплатном
   хостинге, который умеет держать долгоживущий процесс.

Клиент и сервер общаются по **wss://** (WebSocket поверх TLS). Клиент раздаётся по
`https://`, а страница по `https://` **не может** открыть незашифрованный
`ws://` — поэтому серверу обязательно нужен TLS-сертификат. Оба пути ниже это
обеспечивают (Render — сам, Oracle — через Caddy).

Порядок действий: **сначала поднимаем сервер** (получаем его адрес), **потом
собираем и публикуем клиент** с этим адресом.

## 2. Проверка локально (перед облаком)

Убедимся, что всё собирается и запускается на твоём компьютере.

**Сервер:**
```bash
cd путь/к/sheeeeps
dart run bin/server.dart
```
Успех, если в консоли появилось:
```
[…] pasture server up — listening on 0.0.0.0:8080 (protocol v2, 450 sheep, up to 16 shepherds)
[…] health check: http://0.0.0.0:8080/health
```
Проверка «жив ли»: в другом терминале `curl http://localhost:8080/health` → должно
ответить `ok`. Оставь сервер запущенным.

**Клиент (в новом терминале):**
```bash
flutter run -d chrome -t lib/main_pasture.dart --dart-define=PASTURE_URL=ws://localhost:8080
```
Откроется браузер: введи имя → «Играть» → появишься собакой на пастбище. Погоняй
овец, загони всех в вольер — увидишь фейерверк и таймер раунда. Работает локально —
значит код в порядке, дальше только вопрос хостинга.

Останови оба процесса (`Ctrl+C`).

## 3. Сервер, путь A — Render (проще всего, для быстрой пробы)

Render сам соберёт `Dockerfile` из репозитория и выдаст готовый `https/wss` адрес с
TLS — карта не нужна. Честные минусы бесплатного тарифа: сервис **засыпает после 15
минут без запросов** и просыпается ~1 минуту при следующем заходе (состояние раунда
сбрасывается — по ТЗ допустимо), и лимит исходящего трафика всего **5 ГБ/мес**. Этого
хватает на первую пробу с парой человек; для настоящего живого мира бери Oracle
(раздел 4).

1. Зайди на [dashboard.render.com](https://dashboard.render.com) → **New +** →
   **Web Service**.
2. Источник: **Build and deploy from a Git repository** → подключи GitHub → выбери
   репозиторий `sheeeeps`.
3. **Language / Runtime**: Render увидит `Dockerfile` в корне и выберет **Docker**.
   Root directory оставь пустым, ветка `main`.
4. **Instance Type**: выбери **Free**.
5. **Health Check Path**: впиши `/health` (в разделе Advanced). Порт настраивать не
   надо — Render сам передаёт процессу переменную `PORT`, а наш сервер её читает.
6. Region: любой ближайший (например `Frankfurt`).
7. Нажми **Create Web Service**. Первая сборка идёт ~5–10 минут (собирается Docker-образ).

**Как понять, что удалось:** статус сервиса станет **Live** (зелёный), а сверху будет
адрес вида `https://sheeeeps-ТВОЙ.onrender.com`. Открой
`https://sheeeeps-ТВОЙ.onrender.com/health` — должно написать `ok` (первый заход после
паузы может подумать ~минуту, пока сервис просыпается).

**Запиши адрес сервера для клиента:** возьми этот URL и замени `https` на `wss`:
```
wss://sheeeeps-ТВОЙ.onrender.com
```
Это и есть `PASTURE_URL`. Переходи к [разделу 5](#5-клиент--cloudflare-pages).

## 4. Сервер, путь B — Oracle Cloud Always Free (честные 24/7 + 10 ТБ)

Дольше в настройке, но это настоящая всегда-включённая машина: процесс не засыпает,
пастбище живёт круглосуточно, трафика 10 ТБ/мес — с колоссальным запасом. Рекомендую,
если проект «пойдёт» и нужен стабильный мир.

### 4.1. Создать бесплатную ВМ

1. В [консоли Oracle Cloud](https://cloud.oracle.com) → **Menu → Compute → Instances
   → Create instance**.
2. **Image and shape**: жми **Change shape → Ampere** → `VM.Standard.A1.Flex`.
   Поставь 1–2 OCPU и 6–12 GB RAM (в пределах Always Free). Образ оставь
   **Canonical Ubuntu 22.04**.
3. **Networking**: оставь создать новую VCN. Убедись, что стоит галка
   **Assign a public IPv4 address**.
4. **Add SSH keys**: выбери **Generate a key pair for me** и **скачай приватный ключ**
   (понадобится для входа). Или загрузи свой публичный ключ.
5. **Create**. Через минуту у инстанса появится **Public IP address** — запиши его.

### 4.2. Открыть порты 80 и 443

Oracle по умолчанию закрывает всё. Открой HTTP/HTTPS:
1. На странице инстанса → **Virtual Cloud Network** → **Security Lists** → default.
2. **Add Ingress Rules**: Source `0.0.0.0/0`, IP Protocol `TCP`, Destination Port
   `80`. Добавь ещё одно правило для порта `443`.
3. Сохрани.

### 4.3. Зайти на ВМ и поставить Docker + Caddy

```bash
ssh -i путь/к/скачанному_ключу ubuntu@ПУБЛИЧНЫЙ_IP
```
На самой ВМ:
```bash
# Открыть порты и в локальном фаерволе Ubuntu:
sudo iptables -I INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -I INPUT -p tcp --dport 443 -j ACCEPT
sudo netfilter-persistent save   # если команды нет: sudo apt install -y iptables-persistent

# Docker:
curl -fsSL https://get.docker.com | sudo sh

# Caddy (даёт бесплатный TLS-сертификат автоматически):
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update && sudo apt install -y caddy
```

### 4.4. Нужен домен (для TLS)

Let's Encrypt не выдаёт сертификат на голый IP. Проще всего — бесплатный поддомен на
[DuckDNS](https://www.duckdns.org): войди через GitHub, придумай имя (например
`sheeeeps`), впиши **Public IP** своей ВМ, нажми **update**. Получишь адрес вида
`sheeeeps.duckdns.org`.

### 4.5. Собрать и запустить сервер

```bash
sudo apt install -y git
git clone https://github.com/ТВОЙ_ЛОГИН/sheeeeps.git
cd sheeeeps
sudo docker build -t pasture-server .          # ~5–10 минут в первый раз
sudo docker run -d --name pasture --restart=always -p 8080:8080 pasture-server
```
`--restart=always` — это и есть супервизор из ТЗ: контейнер сам поднимется после
падения или перезагрузки ВМ. Проверка: `curl http://localhost:8080/health` → `ok`.

### 4.6. Включить Caddy (wss + TLS)

```bash
sudo nano /etc/caddy/Caddyfile
```
Впиши (замени домен на свой из DuckDNS):
```
sheeeeps.duckdns.org {
	reverse_proxy localhost:8080
}
```
(готовый шаблон лежит в `deploy/Caddyfile` в репозитории.) Затем:
```bash
sudo systemctl restart caddy
```

**Как понять, что удалось:** открой `https://sheeeeps.duckdns.org/health` в браузере
— должно написать `ok`, и замок в адресной строке (валидный TLS). Первый запрос может
занять несколько секунд, пока Caddy получает сертификат.

**Адрес сервера для клиента:**
```
wss://sheeeeps.duckdns.org
```

> **Альтернатива без Docker** — запуск нативного бинарника под systemd (юнит-файл в
> `deploy/pasture-server.service`): установи Flutter на ВМ, `dart compile exe
> bin/server.dart -o /opt/sheeeeps/pasture-server`, создай пользователя `sheeeeps`,
> скопируй юнит в `/etc/systemd/system/`, `sudo systemctl enable --now
> pasture-server`. Docker-путь выше проще и не зависит от архитектуры.

## 5. Клиент — Cloudflare Pages

Теперь, зная адрес сервера (`wss://…`), собираем клиент и публикуем статику. У
Cloudflare Pages **безлимитный трафик** — идеально для раздачи игры.

### 5.1. Собрать клиент с адресом сервера

Подставь свой `wss://…` адрес из раздела 3 или 4:
```bash
# Linux/macOS:
scripts/build-web.sh wss://sheeeeps-ТВОЙ.onrender.com
# Windows PowerShell:
.\scripts\build-web.ps1 wss://sheeeeps-ТВОЙ.onrender.com
```
Появится папка `build/web` — это готовый сайт. Адрес сервера «зашит» внутрь.

### 5.2. Опубликовать на Cloudflare Pages

Самый простой способ — прямая загрузка папки:
1. [dash.cloudflare.com](https://dash.cloudflare.com) → в меню слева **Workers & Pages**
   → **Create** → вкладка **Pages** → **Upload assets**.
2. Назови проект (например `sheeeeps`), перетащи в окно **содержимое папки
   `build/web`** (все файлы, `index.html` должен оказаться в корне).
3. **Deploy site**.

**Как понять, что удалось:** Cloudflare выдаст адрес вида
`https://sheeeeps.pages.dev`. Открой его — увидишь экран ввода имени. Введи имя →
«Играть» → должен появиться на пастбище (в правом верхнем углу — таблица «Загнали»,
сверху — таймер раунда и рекорд дня). **Это и есть публичная ссылка на игру.**

## 6. Как обновлять после правок кода

**Клиент** (любые правки в `lib/`, кроме серверных):
```bash
scripts/build-web.sh wss://ТВОЙ_адрес_сервера        # пересобрать
```
Затем снова **Upload assets** в том же проекте Cloudflare Pages (или, если проект
подключён к GitHub, просто `git push` — Cloudflare пересоберёт сам). Игрокам, возможно,
понадобится **Ctrl+F5**, чтобы сбросить кэш браузера.

**Сервер:**
- **Render**: `git push` в `main` — Render автоматически пересоберёт и передеплоит. Или
  в карточке сервиса кнопка **Manual Deploy → Deploy latest commit**.
- **Oracle (Docker)**:
  ```bash
  cd sheeeeps && git pull
  sudo docker build -t pasture-server .
  sudo docker rm -f pasture && sudo docker run -d --name pasture --restart=always -p 8080:8080 pasture-server
  ```

> Меняешь адрес сервера? Тогда пересобери **и** клиент (`build-web`) с новым
> `PASTURE_URL`, иначе он будет стучаться на старый.

## 7. Логи и перезапуск

**Render:** вкладка **Logs** в карточке сервиса. Перезапуск — **Manual Deploy** или
меню **⋯ → Restart service**.

**Oracle (Docker):**
```bash
sudo docker logs -f pasture        # смотреть логи в реальном времени
sudo docker restart pasture        # перезапустить
sudo docker rm -f pasture          # остановить и удалить контейнер
```

Что пишет сервер в лог (по ТЗ): старт; каждое подключение/отключение
(`join`/`leave`); завершение раунда с временем и рекордом дня; и раз в 10 секунд —
перцентили времени тика и число игроков, например:
```
[…] tick p50=0.18ms p95=0.33ms p99=0.51ms max=1.2ms n=600 players=7
```
Тик должен держаться заметно ниже 10 мс даже при 16 собаках и 500 овцах (замерено:
p99 ≈ 0.4 мс на обычной машине).

## 8. Если не работает

- **Экран «Подключение к пастбищу…» висит вечно / «Не удалось подключиться».**
  Клиент не достучался до сервера. Проверь: (1) открой `https://адрес-сервера/health`
  — если не `ok`, проблема на стороне сервера; (2) клиент собран с правильным
  `PASTURE_URL` (пересобери `build-web` с точным `wss://…`); (3) адрес именно `wss://`,
  не `ws://`.

- **В консоли браузера (F12) `Mixed Content` / `insecure WebSocket`.** Страница по
  `https`, а сервер по `ws://` без TLS. Нужен `wss://`: на Render он есть из коробки,
  на Oracle — подними Caddy (раздел 4.6). Голый IP без домена не годится — нужен домен
  (DuckDNS).

- **`wss` не коннектится, хотя `/health` по https отвечает.** Значит TLS есть, но
  апгрейд WebSocket не проходит. На Oracle убедись, что в `Caddyfile` именно
  `reverse_proxy localhost:8080` (Caddy проксирует WebSocket сам). Перезапусти Caddy.

- **Порт закрыт фаерволом (Oracle).** Две стены: (1) **Security List** в OCI (порты
  80/443, раздел 4.2) и (2) локальный `iptables` на ВМ (раздел 4.3). Нужно открыть
  обе. Проверка снаружи: `curl -I https://твой-домен/health`.

- **Процесс умирает после закрытия SSH.** Не запускай сервер «руками» в SSH-сессии.
  Используй `docker run -d --restart=always …` или systemd-юнит — они живут отдельно
  от твоего терминала и перезапускаются сами.

- **Браузер показывает старый клиент после обновления.** Кэш. Жёсткая перезагрузка
  **Ctrl+F5** (Cmd+Shift+R на macOS). Cloudflare отдаёт новую сборку сразу, кэширует
  браузер.

- **Render: первый заход после паузы долго грузится (~минуту).** Бесплатный тип
  «засыпает» после 15 минут без запросов и просыпается при первом заходе. Это нормально.
  Нужен always-on без пауз — путь Oracle (раздел 4).

- **Render: «трафик закончился» / сервис остановлен.** На Free всего 5 ГБ исходящего в
  месяц (см. раздел 9) — на активную игру с несколькими людьми это мало. Как упрёшься —
  переезжай на Oracle (10 ТБ).

## 9. Бесплатные лимиты и трафик

Цифры проверены в июле 2026 (условия у хостеров меняются — сверяйся по ссылкам).

**Клиент — Cloudflare Pages:** трафик **безлимитный** на всех тарифах, до **500
сборок/мес** на Free. Для раздачи статики ограничений по трафику фактически нет.
([Cloudflare Pages limits](https://developers.cloudflare.com/pages/platform/limits))

**Сервер — Render Free:** `0.1 CPU / 512 MB RAM`, **5 ГБ исходящего трафика/мес**,
засыпает после 15 минут простоя, просыпается ~1 мин. 5 ГБ при стриминге снапшотов — это
всего ~23 «клиенто-часа» в месяц, так что Render годится лишь на короткую пробу.
([Render Free docs](https://render.com/docs/free))

> Koyeb, который был здесь раньше как удобный бесплатный вариант, с февраля 2026
> недоступен для новых пользователей (куплен Mistral).

**Сервер — Oracle Always Free:** `VM.Standard.A1.Flex` до 2 ядер / 12 ГБ RAM (лимиты
урезаны с июня 2026, но всё ещё с запасом), **10 ТБ исходящего трафика/мес**, работает
постоянно и бесплатно.
([cloudpricecheck](https://cloudpricecheck.com/free-tier/oracle),
[Oracle Always Free docs](https://docs.oracle.com/en-us/iaas/Content/FreeTier/freetier_topic-Always_Free_Resources.htm))

**Бюджет трафика игры.** Один снапшот на 500 овец ≈ 2.7 КБ, шлётся 20 раз/сек ⇒
≈ **55 КБ/с на игрока**. Сервер шлёт снапшот каждому подключённому, значит суммарный
исходящий ≈ `игроки × 55 КБ/с`. Прикидка на месяц:
- 1 игрок за ~1.5 часа игры сжигает почти весь месячный лимит **Render** (5 ГБ).
- 4 игрока по 3 часа в день ⇒ ≈ **70 ГБ/мес** — только Oracle.
- 16 игроков непрерывно ⇒ ≈ 76 ГБ/**день** — только Oracle (10 ТБ).

**Вывод.** **Render** — чтобы за 10 минут увидеть игру онлайн и показать паре друзей
(но 5 ГБ/мес закончатся быстро и сервис засыпает). Для настоящего живого мира 24/7 и
спокойствия по трафику — **Oracle Always Free** (рекомендую). Когда пастбище пустует,
снапшоты не шлются вовсе (0 игроков ⇒ 0 трафика), так что реальная цифра обычно сильно
ниже прикидки.

**Почему не Fly.io / Koyeb:** у Fly.io бесплатный тариф закрыт (с 2024, теперь от
~$2/мес), а Koyeb с февраля 2026 куплен Mistral и закрыл бесплатную регистрацию. Render
формально жив, но 5 ГБ/мес — только на короткую пробу; всерьёз бесплатный 24/7 остаётся
**Oracle Always Free**.
([Fly.io](https://www.saaspricepulse.com/blog/flyio-free-tier-2026),
[Render Free](https://render.com/docs/free))
